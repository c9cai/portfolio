MMS Restful Android App

2016 Summer project I worked on during my internship at Sansay.

OBJECTIVE

The objective of the project was to develop a mobile application that could pull information from their existing MMS (database monitoring) interface. Previously, every web application was run off of SOAP ws, but we ran into a few problems, and proceeded to try out RESTful ws instead.

-------------------------------------------------------------------------------------------------------------

DATA ANALYSIS

The data that we wanted to retrieve was stored in various SQL tables. I compiled a list of the ones that I needed and created a new class to organize and store all of them.

-------------------------------------------------------------------------------------------------------------

CLIENT SERVER INTERACTION

The primary client server interaction would work as follows: The client would first initiate an https connection with the server, using the ClientKeystore file to authenticate. After a secure connection is achieved, depending on user input, the client will send a get request with the specified parameters, which will be processed by the correct resource file on the server side. The server will use custom SQL queries to populate the android data object, and then send a reply back to the client. This reply will then be parsed and charted.

-------------------------------------------------------------------------------------------------------------

ACTIVITIES

Now that I had the backend done, I had to work on the front end portion of the app. The first thing I had to understand was how activities worked on Android Applications.


An activity represents a single screen with a user interface. Each activity has a java file for backend logic, and a xml file to define on screen items(views) in the activity. In addition, each activity has some methods that must be overridden, onCreate(), on onPause(), etc, that correspond to different parts of the activity lifecycle:

-------------------------------------------------------------------------------------------------------------

INTENTS

Another important concept I had to learn was Intents. To summarize, Intents are used to start another activity from within a current one. Intents can be explicit or implicit. Explicit intents are used to start a specific activity (used in this android app) by specifying the class name in the constructor Implicit intents are used to start a type of activity or function, i.e display a web page or show a location on google maps. Each intent can also have “Extras” which essentially act as a key to value Map in order to pass objects/information between activities.

-------------------------------------------------------------------------------------------------------------

USER INTERFACE

Now that I had learned the underlying processes of Android apps, the last step was implementing UI. Each on screen item is called a View - an object that draws something on the screen that the user can interact with. These are contained a ViewGroups, which hold other View/ViewGroups to define the layout. Views can be buttons, textViews, listViews, etc, and need to be defined in the Activity’s xml file. To populate/interact with the views, they are inflated (findViewById method, identified by android:id in xml doc) from the Activities java file, and then built in methods can be used to set Text, or update other properties.

Music App Redesign

Final project for COGS187A Fall 2016

For my final project in COGS187A, I worked with a partner on a music app redesign, to make it easier for users to create and manage their playlists. Because it was a 2 man group, we didn't really split up the work, and we basically worked together on everything from start to finish.

-------------------------------------------------------------------------------------------------------------

INTERVIEWS

The first step was conducting user interviews to find out what people liked and didn't like about whatever music streaming service they used. Here are some of the questions we asked:

What we found out from these interviews was that most people liked using Spotify and Soundcloud the most, with spotify being the most popular. This seemed to be due to a combination of simplicity and utility. Out of all the apps, spotify had the best utility to find new songs for users to add to their playlists, which was a big plus. Following our interviews, we developed a few user scenarios based on responses we received.

  1. Creating a playlist for the gym.
  2. Organizing playlist based on recently added.
  3. Viewing friend's playlists to find what they're listening to.
  4. Creating a playlist from an artist's album.
  5. Creating a playlist for a party

-------------------------------------------------------------------------------------------------------------

COMPETETIVE ANALYSIS

In order to better understand music streaming apps do worse or better than their peers, we performed a competetive analysis between 3 of the biggest apps: Soundcloud, Spotify, and Apple Music. Here's an example of a comparison that we didn between Spotify and SoundCloud regarding the ability to organize music on your playlist. (e.g by date,artist...)

After performing 9 of these, we had a good idea of what we liked and didn't like about each app.

-------------------------------------------------------------------------------------------------------------

CREATING SITEMAPS

The next step was to create sitemaps of the 3 different services, in order to see the different layouts of the apps, and see what was similar and what was different. By isolating and visualizing these differences, it became a lot more apparant what was working well and what wasn't. Here's the competetive analysis we did of SoundCloud.

-------------------------------------------------------------------------------------------------------------

WIREFRAME REDESIGN

The last step was actually redesigning the music app into a wireframe, which we did using a combination of Invision and Sketch. Here are some of the main changes we implemented.

Out of all of the different kinds of home pages, we definitely liked SoundClouds the most, because it didn't overwhelm the user with so many different suggestions, but instead had a friend feed, kind of like Facebook for music. We felt that Apple Music and Spotify's home page functionality could be implemented in the Browse tab, so we based our Home tab off of SoundCloud's.

Out of all of the Browse options, most people reported that they liked Spotify's the most. However, when we followed up by asking how many of those options people actually used, most people reported that they only looked at the popular playlists. We felt like we could optimize the Browse tab by combining it with single song suggestions, like SoundCloud's discover feature. That way there would be a good middle ground between having too much, too broad of a suggestion category, and too narrow of one.

Another tab that we changed was the home Tab. Our's is roughly the same as Spotify's but we moved the user profile with followers/following to the front. We want to remind the user that music is a social thing, and to suggest that they look at what their friends or followers are listening to occasionally.
